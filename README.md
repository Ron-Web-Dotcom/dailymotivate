### Architecture Overview

**DailyMotivate** follows a **Service-Oriented Architecture** with clear separation of concerns:

1. **Presentation Layer** (`lib/presentation/`): UI components and screens
2. **Service Layer** (`lib/services/`): Business logic and data management
3. **Core Layer** (`lib/core/`): Utilities, error handling, and app lifecycle
4. **Theme Layer** (`lib/theme/`): Design system and styling
5. **Widgets Layer** (`lib/widgets/`): Reusable UI components

**Key Design Patterns:**
- **Singleton Services**: All services use singleton pattern for global access
- **Provider Pattern**: Theme management with ChangeNotifier
- **Repository Pattern**: Data access through service layer
- **Factory Pattern**: AI service selection (OpenAI vs Gemini)

## ğŸ”’ Security Notes

### Data Privacy

**DailyMotivate** is designed with privacy-first principles:

1. **Minimal Data Collection**:
   - Email address (only for cloud sync authentication)
   - Favorite quotes (stored locally and optionally in cloud)
   - App usage preferences (theme, notification settings)

2. **No PII in Logs**:
   - All logging is sanitized to remove personally identifiable information
   - API keys and tokens are redacted from error messages
   - Email addresses are masked in logs

3. **Secure Storage**:
   - Local data stored using Flutter's `SharedPreferences` (encrypted on iOS)
   - Cloud data protected by Supabase Row Level Security (RLS) policies
   - API keys never stored in source code (compile-time constants only)

4. **HTTPS Only**:
   - All network requests use HTTPS/TLS 1.2+
   - Cleartext traffic disabled on Android (`usesCleartextTraffic="false"`)
   - Certificate pinning for production (optional)

### Authentication

- **Google Sign-In**: OAuth 2.0 flow with secure token handling
- **Supabase Auth**: JWT tokens with automatic refresh
- **No Password Storage**: All authentication handled by third-party providers

### API Key Security

**CRITICAL**: Never commit API keys to version control
